# 写在理解 vLLM 量化之前，从 FP32 到 FP4：大模型精度演进的"瘦身简史"

> 低精度格式随大模型发展持续演进——从 FP32 到 FP16/BF16，从 INT8/INT4 到 FP8，再到 FP4 混合精度，甚至未来向1/2bit极低比特演进。每一次精度降低，背后都是一场"够用就行"与"不能糊"的博弈。

---

笔者写在前面的话，在正式进入介绍 vLLM 推理系统的量化之前，还是先了解一下 LLM 大模型和量化精度的演进史，更容易理解量化，理解量化，更好的理解注意力机制的工程实现基础，也更加容易理解 vLLM 推理系统有关量化的实现。

### 引言

如果你开了一家跨世纪的百年连锁餐厅，菜单上的价格从上个世纪的精确到分（FP32，至少80后的尾巴90后的老大那一代人的记忆中还有1分钱、2分钱、5分钱硬币），变成精确到角（FP16），再变成整数（INT8），最后变成"十元一档"（FP4，时代变迁，物价逐年通货膨胀，90年代1角钱还能买到几个包子，现在要吃一顿早餐也得10元左右了）。每次降价精度（提升物价，菜价，包子价从5角一个到1元一个再到3元一个），你都在问同一个问题：**顾客能接受吗？**

大模型的精度演进也是一样。过去十年，深度学习的数值格式经历了五次重大跃迁，每次跃迁的驱动力都不同——有时是训练速度，有时是推理显存，有时是硬件换代。本文将按时间线拆解每次跃迁的**技术原理、驱动逻辑和取舍智慧**。

---

### 一、起点：FP32 统治的时代（2012-2017）

##### 1.1 FP32 是什么

FP32（32-bit 浮点数）是 IEEE 754 标准定义的单精度浮点格式：

```
FP32: 1 符号位 + 8 指数位 + 23 尾数位 = 32 bits
      ┌───┬────────────┬─────────────────────────┐
      │ S │ EEEEEEEE   │ MMMMMMMMMMMMMMMMMMMMMMM  │
      └───┴────────────┴─────────────────────────┘
       1b    8b                    23b

表示范围: ±3.4 × 10³⁸
精度:     ~7 位有效十进制数字
```

FP32 的 23 位尾数意味着它可以区分约 1600 万个不同的值——对于大多数计算来说绰绰有余。在 2012 年 AlexNet 引爆深度学习之后的五年里，FP32 是所有框架和硬件的唯一选择。

##### 1.2 为什么当时不需要低精度

| 因素 | 2012-2017 的现实 | 对精度的要求 |
|:---|:---|:---|
| 模型规模 | AlexNet 60M 参数，VGG 138M | FP32 绰绰有余 |
| 显存压力 | 一张 K80 有 12GB，够用 | 不紧迫 |
| 训练稳定性 | 梯度数值范围大，FP32 是安全选择 | 不能降 |
| 硬件支持 | GPU 只有 FP32 计算单元 | 没有选择 |

> 笔者注：FP32 时代并非没有低精度的尝试。2015 年 Google 就在 TPU v1 上用了 INT8 推理，但那是专用硬件的特殊路径，对 GPU 生态没有影响。低精度真正成为主流，要等到 GPU 硬件原生支持。

### 二、第一次跃迁：FP16 与 BF16（2017-2022）

##### 2.1 驱动力：训练速度

2017 年前后，模型规模开始膨胀——ResNet、Transformer、BERT 相继出现，训练时间从小时级飙升到天级甚至周级。FP32 的计算单元利用率低，成为训练速度的瓶颈。

GPU 厂商给出的答案是：**用更低的精度换更快的计算**——FP16 的计算吞吐是 FP32 的 2-4 倍。

##### 2.2 FP16：精度不够，溢出来凑

```
FP16: 1 符号位 + 5 指数位 + 10 尾数位 = 16 bits
      ┌───┬───────┬──────────────┐
      │ S │ EEEEE │ MMMMMMMMMM   │
      └───┴───────┴──────────────┘
       1b    5b         10b

表示范围: ±65504
精度:     ~3 位有效十进制数字
```

FP16 把 FP32 的 32 位砍了一半——指数位从 8 减到 5，尾数位从 23 减到 10。问题立刻暴露：

| 问题 | 原因 | 后果 |
|:---|:---|:---|
| **上溢** | 最大值只有 65504，梯度稍大就 Inf | 训练崩溃 |
| **下溢** | 最小正值约 6×10⁻⁸，小梯度直接变零 | 梯度消失 |
| **精度损失** | 10 位尾数只能区分 1024 个值 | 累加误差 |

解决方案是 **混合精度训练（AMP）**：前向传播用 FP16 加速，但维护一份 FP32 的"主权重"做梯度更新。就像用粗笔快速画草图（FP16 前向），但用细笔精修定稿（FP32 更新）。

```
混合精度训练流程:
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│ FP32 主权重   │────→│ FP16 副本     │────→│ FP16 前向    │
│ (精修定稿)    │     │ (快速草图)    │     │ (快速计算)    │
└──────────────┘     └──────────────┘     └──────────────┘
       ↑                                        │
       │            ┌──────────────┐     ┌──────┴───────┐
       └────────────│ FP32 梯度更新 │←────│ FP16 梯度    │
                    │ (精修定稿)    │     │ Loss Scale   │
                    └──────────────┘     └──────────────┘
```

**Loss Scaling** 是混合精度训练的关键技巧：把 loss 乘一个大的缩放因子（如 2¹⁶），让小梯度在 FP16 范围内不致下溢为零，更新前再缩放回来。

##### 2.3 BF16：Google 的"指数位不砍"方案

FP16 的溢出问题太烦了。Google 在 2018 年提出了 BF16（Brain Float 16），核心思路：**砍尾数保指数**。

```
FP32:  S EEEEEEEE MMMMMMMMMMMMMMMMMMMMMMM   (1+8+23)
BF16:  S EEEEEEEE MMMMMMM                    (1+8+7)
FP16:  S EEEEE   MMMMMMMMMM                  (1+5+10)

对比:
         指数位    尾数位    表示范围         精度
FP32:     8        23       ±3.4×10³⁸       ~7位十进制
BF16:     8         7       ±3.4×10³⁸       ~2位十进制  ← 和 FP32 同范围！
FP16:     5        10       ±65504           ~3位十进制
```

BF16 保留了 FP32 的全部 8 位指数，所以表示范围和 FP32 完全相同——不需要 Loss Scaling，不会溢出。代价是尾数只有 7 位，精度比 FP16 还低，但深度学习的梯度更新对范围敏感远大于对精度敏感。

| 格式 | 表示范围 | 精度 | 需要 Loss Scale | 训练稳定性 | 推理精度 |
|:---|:---|:---|:---|:---|:---|
| FP32 | ±3.4×10³⁸ | 最高 | 否 | 最好 | 最好 |
| FP16 | ±65504 | 中等 | 是 | 需要调参 | 好 |
| BF16 | ±3.4×10³⁸ | 最低 | 否 | 接近 FP32 | 略差 |

> 笔者注：BF16 最初只在 Google TPU 上使用，2020 年 NVIDIA A100（Ampere）原生支持 BF16 后，BF16 迅速成为大模型训练的事实标准。现在几乎所有大模型（GPT-4、Llama、DeepSeek）都以 BF16 训练和存储。

##### 2.4 FP16/BF16 的历史意义

FP16 和 BF16 标志着深度学习从"精度至上"转向"够用就好"——这是精度演进的核心哲学。关键认知转变：

**深度学习的数值不需要均匀精度**——梯度需要大范围（BF16 的 8 位指数），权重需要适中精度（7-10 位尾数就够了），而中间激活值甚至可以更粗。这个认知为后续所有低精度格式铺平了道路。

### 三、第二次跃迁：INT8 与 INT4（2022-2024）

##### 3.1 驱动力：推理显存

2022-2023 年，LLM 爆发。GPT-3 175B、Llama-65B 等模型的推理面临一个残酷现实：

```
Llama-65B 推理显存需求:
  FP16: 65B × 2 bytes = 130 GB  → A100-80GB 装不下，需要 2 卡
  INT8: 65B × 1 byte  = 65 GB   → 单卡 A100-80GB 可跑
  INT4: 65B × 0.5 byte = 32.5 GB → 单卡 A100-40GB 也能跑！
```

训练时我们关心速度，推理时我们更关心**能不能装下**。INT8/INT4 量化的驱动力是显存容量，而非计算速度。

##### 3.2 INT8：整数世界的"够用"标准

INT8 是 8 位有符号整数，范围 [-128, 127]。和浮点数不同，整数没有指数位——所有 256 个值均匀分布在量化范围内。

```
FP16 浮点数: 值的分布不均匀，小值附近密集，大值附近稀疏
  |···························              ··········|
  0                                      65504

INT8 整数量化: 值均匀分布，每个区间大小相同
  |·  ·  ·  ·  ·  ·  ·  ·  ·  ·  ·  ·  ·  ·  ·  ·|
  -128                                         127
```

INT8 量化的核心问题：**浮点权重怎么映射到 256 个整数？** 这就是下一篇 vLLM量化文章中将要讨论的对称/非对称量化和 Per-tensor/Per-channel/Per-block 粒度选择。

##### 3.3 INT4：极限压缩的代价

INT4 只有 16 个值（[-8, 7] 或 [0, 15]），压缩率高达 4 倍（相对 FP16），但代价也很大：

```
INT4 的困境: 16 个值能表达多少信息？

权重分布 (FP16):          INT4 量化后:
    ╱╲                       │ │
   ╱  ╲                     │ │ │
  ╱    ╲                    │ │ │ │
 ╱      ╲                  │ │ │ │ │
╱        ╲                │ │ │ │ │ │
──────────────            ─────────────
连续分布，细节丰富         只有 6-7 个离散档位

问题: 重要的权重（红色区域）和次要的权重（蓝色区域）挤在同一个档位
      → AWQ 的解法: 保护重要权重，用更精细的 scale
      → GPTQ 的解法: 用 Hessian 信息逐列最优量化
```

| 对比 | INT8 | INT4 |
|:---|:---|:---|
| 压缩比（vs FP16） | 2× | 4× |
| 可表达值数 | 256 | 16 |
| 精度损失 | 小（多数场景可接受） | 大（需要算法补偿） |
| 算法复杂度 | 简单缩放 | 需要 AWQ/GPTQ 等高级算法 |
| 适用场景 | 生产环境通用 | 显存极度紧张或吞吐优先 |

##### 3.4 Weight-Only vs Weight-Activation 量化

INT8/INT4 量化有一个关键区分：**只量化权重，还是权重和激活值都量化？**

```
Weight-Only 量化 (W8A16 / W4A16):
  权重: FP16 → INT8/INT4 (存储在低精度)
  激活: 保持 FP16 (计算时反量化回 FP16)
  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ INT4 权重 │───→│ 反量化    │───→│ FP16 矩阵│
  │ (存储小)  │    │ (即时还原)│    │ 乘法     │
  └──────────┘    └──────────┘    └──────────┘

  收益: 显存降低 2-4×，权重搬运量降低 2-4×
  代价: 计算仍为 FP16，反量化有额外开销

Weight-Activation 量化 (W8A8):
  权重: FP16 → INT8
  激活: FP16 → INT8 (计算也在 INT8 下进行)
  ┌──────────┐    ┌──────────┐
  │ INT8 权重 │    │ INT8 激活│
  └─────┬────┘    └─────┬────┘
        └───────┬───────┘
                ▼
          INT8 矩阵乘法 (硬件原生加速)
                │
                ▼
          INT8 → FP16 输出

  收益: 显存 + 计算 + 搬运全面加速
  代价: 激活值量化可能导致精度损失更大
```

| 模式 | 缩写 | 计算精度 | 适用场景 |
|:---|:---|:---|:---|
| Weight-Only INT8 | W8A16 | FP16 | 通用推理，精度优先 |
| Weight-Only INT4 | W4A16 | FP16 | 显存紧张，吞吐优先 |
| Weight+Activation INT8 | W8A8 | INT8 | 有 INT8 Tensor Core 的硬件 |
| Weight+Activation FP8 | W8A8 (FP8) | FP8 | Hopper/Blackwell |

> 笔者注：Weight-Only 量化之所以流行，是因为 LLM 推理的 Decode 阶段是**内存带宽瓶颈**（memory-bound）——每生成一个 Token，都要把全部权重从 HBM 搬到 SM。更小的权重 = 更少的搬运 = 更快的生成。计算本身不是瓶颈，所以激活值保持 FP16 不影响速度。

##### 3.5 INT8/INT4 的历史意义

INT8/INT4 标志着精度演进从"训练驱动"转向"推理驱动"。核心认知转变：

**推理时权重是静态的，可以用更激进的压缩；激活值是动态的，需要更保守的策略。** 这催生了 Weight-Only 量化范式——只压缩权重，保持激活值精度。AWQ 保护重要权重、GPTQ 用 Hessian 最优量化，都是在"16 个值够不够"的极限条件下做算法补偿。

### 四、第三次跃迁：FP8（2024-2025）

##### 4.1 驱动力：硬件原生支持

INT8/INT4 的量化需要算法补偿（AWQ/GPTQ），增加了部署复杂度。如果硬件原生支持一种比 FP16 更小、但比 INT8 更聪明的格式呢？

NVIDIA 在 Hopper 架构（H100, 2023）中引入了 FP8 Tensor Core——**硬件原生支持 FP8 矩阵乘法**，不需要反量化，不需要算法补偿，开箱即用。

##### 4.2 FP8 的两种格式：E4M3 和 E5M2

FP8 不是一种格式，而是两种——分别优化前向和反向传播：

```
FP8 E4M3 (前向传播，精度优先):
  ┌───┬──────┬─────┐
  │ S │ EEEE │ MMM │    1+4+3 = 8 bits
  └───┴──────┴─────┘
  表示范围: ±448
  精度:     3 位尾数，可区分 8 个不同值
  特点:     无 Inf，精度更高

FP8 E5M2 (反向传播，范围优先):
  ┌───┬───────┬────┐
  │ S │ EEEEE │ MM │    1+5+2 = 8 bits
  └───┴───────┴────┘
  表示范围: ±57344
  精度:     2 位尾数，可区分 4 个不同值
  特点:     支持 Inf/NaN，范围更大
```

为什么前向和反向需要不同格式？因为它们的数值分布不同：

```
前向传播的权重和激活值:
  分布相对集中，数值范围窄
  → 需要更多尾数位来区分相近的值
  → E4M3: 3 位尾数，范围 ±448 够用

反向传播的梯度:
  分布范围广，偶尔出现极端值
  → 需要更多指数位来覆盖大范围
  → E5M2: 5 位指数，范围 ±57344
```

| 格式 | 指数位 | 尾数位 | 范围 | 精度 | 用途 |
|:---|:---|:---|:---|:---|:---|
| FP8 E4M3 | 4 | 3 | ±448 | 较高 | 前向权重/激活 |
| FP8 E5M2 | 5 | 2 | ±57344 | 较低 | 反向梯度 |
| INT8 | — | — | [-128, 127] | 均匀 | 通用量化 |

##### 4.3 FP8 vs INT8：浮点回归

FP8 和 INT8 都是 8 位，但哲学截然不同：

```
INT8: 均匀量化
  ││││││││││││││││││││││││││││││││││││││││││││││││││││
  -128                                           127
  每个区间大小相同 → 小值和大值的精度一样
  问题: 深度学习的权重分布不均匀，小值附近需要更高精度

FP8: 浮点量化
  │···························              ··········│
  0                                          448
  小值附近密集，大值附近稀疏 → 自适应精度
  优势: 不需要校准数据集找最优缩放因子，天然适配权重分布
```

| 对比 | INT8 | FP8 |
|:---|:---|:---|
| 值的分布 | 均匀（256 个等距值） | 非均匀（小值密集，大值稀疏） |
| 量化方式 | 需要校准找 scale/zero_point | 只需找 scale，无需 zero_point |
| 硬件支持 | 通用（所有 GPU） | Hopper+（H100/B200） |
| 训练支持 | 否（仅推理） | 是（E4M3 前向 + E5M2 反向） |
| 精度 | 需要算法补偿（AWQ/GPTQ） | 通常无需补偿 |
| 部署复杂度 | 高（选算法、调参数） | 低（硬件原生，开箱即用） |

> 笔者注：FP8 的"浮点回归"是精度演进的一个有趣转折——从浮点（FP32/FP16）到整数（INT8/INT4），又回到浮点（FP8）。区别在于 FP8 不是简单的"砍半 FP16"，而是专门为深度学习设计的 8 位浮点格式，兼顾了浮点的自适应精度和 8 位的存储效率。

##### 4.4 FP8 的历史意义

FP8 标志着精度演进从"算法补偿"转向"硬件原生"。核心认知转变：

**最好的量化不是用算法弥补精度损失，而是让硬件直接支持一种"够聪明"的低精度格式。** FP8 的浮点分布天然适配深度学习的数值特征，不需要 AWQ/GPTQ 那样的算法补偿，也不需要校准数据集——H100 上 `--quantization fp8` 一行命令搞定。

### 五、第四次跃迁：FP4 与混合精度（2025+）

##### 5.1 驱动力：模型规模再上台阶

2025 年，万亿参数模型（DeepSeek-R1 671B MoE、Llama-4 400B+）成为常态。即使 FP8 也无法让这些模型在合理数量的 GPU 上运行：

```
DeepSeek-R1 671B MoE (激活 37B 参数):
  FP16: 671B × 2 = 1342 GB  → 需要 17 张 H100
  FP8:  671B × 1 = 671 GB   → 需要 9 张 H100
  FP4:  671B × 0.5 = 335 GB → 需要 5 张 H100
```

FP4 是 NVIDIA Blackwell 架构（B200, 2025）引入的 4 位浮点格式。

##### 5.2 FP4：4 位浮点的极限

FP4 只有 4 位——1 位符号 + 2 位指数 + 1 位尾数：

```
FP4: ┌───┬────┬───┐
     │ S │ EE │ M │    1+2+1 = 4 bits
     └───┴────┴───┘

可表示的值 (E2M1, 假设 bias=1):
  符号  指数  尾数    值
   0     00    0     0.0
   0     00    1     0.5
   0     01    0     1.0
   0     01    1     1.5
   0     10    0     2.0
   0     10    1     3.0
   0     11    0     4.0
   0     11    1     6.0
   ... (负数对称)

总共只有 6 个正数值 + 6 个负数值 + 零 = 13 个非零值！
```

4 位浮点只有 **13 个非零值**——比 INT4 的 16 个值还少，而且分布不均匀。这意味着 FP4 不能单独使用，必须配合更高精度的缩放因子。

##### 5.3 混合精度：FP4 不单打独斗

FP4 的真正用法是**混合精度**——权重用 FP4 存储，但每个 block 配一个高精度（FP32 或 FP16）的缩放因子：

```
FP4 混合精度存储格式 (Blackwell):

每个 128×128 的权重块:
┌─────────────────────────────────────────────┐
│ 128×128 = 16384 个 FP4 值                   │  每个 4 bits
│ (量化后的低精度权重)                          │  共 8 KB
├─────────────────────────────────────────────┤
│ 1 个 FP32 缩放因子                           │  每个 32 bits
│ (该块的全局缩放)                              │  共 4 bytes
└─────────────────────────────────────────────┘

存储开销: 8 KB + 4 bytes ≈ 8 KB
等效精度: 4 bits + 32-bit scale ≈ 接近 INT8 的表达能力
```

为什么一个 FP32 的 scale 就能救活 FP4？因为每个 block 内的权重分布通常很集中——大部分值落在一个小范围内，一个精确的 scale 就能把 FP4 的 13 个值"对准"到正确的位置。

```
比喻: FP4 混合精度就像"分段线性近似"

原始函数: y = f(x)  (FP16 精度)
                    ╱╲
                   ╱  ╲
                  ╱    ╲
                 ╱      ╲

FP4 混合精度: 把 x 轴分成多个小段，每段用一个 scale + 13 个离散值
                    ┌──┐  ┌──┐  ┌──┐  ┌──┐
                    │╱ │  │╲ │  │╲ │  │╲ │
                    └──┘  └──┘  └──┘  └──┘
                    s₁    s₂    s₃    s₄   ← 每段独立的 FP32 scale

段越短（block 越小），近似越精确
```

##### 5.4 精度演进的混合模式

当前大模型推理中，多种精度格式**同时存在**，各司其职：

```
一个典型的 FP8/FP4 混合推理管线:

┌──────────────────────────────────────────────────────┐
│                    模型权重存储                        │
│                                                      │
│  注意力层权重: FP4 + FP32 block scale (最省显存)       │
│  FFN 层权重:   FP8 E4M3 (精度更敏感)                  │
│  Embedding:    BF16 (不可量化)                        │
│  LayerNorm:    BF16 (不可量化)                        │
│  KV Cache:     FP8 E4M3 (省显存，精度够用)            │
│                                                      │
│  ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐                │
│  │FP4  │  │FP8  │  │BF16 │  │FP8  │                │
│  │Attn │  │FFN  │  │Emb  │  │KV   │                │
│  └──┬──┘  └──┬──┘  └──┬──┘  └──┬──┘                │
│     │        │        │        │                     │
└─────┼────────┼────────┼────────┼─────────────────────┘
      │        │        │        │
      ▼        ▼        ▼        ▼
┌──────────────────────────────────────────────────────┐
│                 计算内核 (GPU SM)                      │
│                                                      │
│  FP4 权重 → 反量化到 FP16 → 与 FP16 激活值做矩阵乘法   │
│  FP8 权重 → FP8 Tensor Core 原生计算                  │
│  BF16 权重 → BF16 Tensor Core 原生计算                │
│  FP8 KV Cache → FlashAttention 原生支持               │
└──────────────────────────────────────────────────────┘
```

| 组件 | 推荐精度 | 原因 |
|:---|:---|:---|
| 注意力 QKV/O 权重 | FP4 / FP8 | 权重量大，精度容忍度高 |
| FFN gate/up/down 权重 | FP8 | MoE 场景下权重占主导，FP4 可进一步压缩 |
| Embedding 表 | BF16 | 词表查表，量化会破坏语义 |
| LayerNorm / RMSNorm | BF16 | 归一化对精度敏感 |
| KV Cache | FP8 E4M3 | 显存大户，FP8 省一半且精度够用 |
| 激活值 | FP16 / BF16 | 动态范围大，量化风险高 |

##### 5.5 FP4/混合精度的历史意义

FP4 标志着精度演进进入"混合编排"阶段。核心认知转变：

**不存在一种"万能精度"——不同组件、不同层、甚至同一层的不同 block，需要不同的精度。** 混合精度不是权宜之计，而是最优策略：在精度最不敏感的地方用最激进的压缩（FP4），在精度最敏感的地方保持高精度（BF16），中间地带用 FP8。

回顾四次跃迁，背后的驱动力清晰可见：

                             推理带宽                   推理显存

关键认知:    关键认知:    关键认知:      关键认知:    关键认知:

| 跃迁 | 时间 | 从→到 | 驱动力 | 核心突破 | 关键技术 |
|:---|:---|:---|:---|:---|:---|
| 第一次 | 2017 | FP32→FP16/BF16 | 训练速度 | 混合精度训练 + Loss Scaling | AMP, BF16 |

### 七、精度选择决策指南

面对这么多精度格式，实际部署时怎么选？

##### 7.1 根据硬件选择

| 硬件 | 可用精度 | 推荐方案 |
|:---|:---|:---|
| V100 (Volta) | FP16, INT8 | FP16 推理，INT8 Weight-Only |
| A100 (Ampere) | FP16, BF16, INT8 | BF16 推理，INT8/INT4 Weight-Only |
| H100 (Hopper) | FP16, BF16, INT8, FP8 | **FP8 推理**（开箱即用），KV Cache FP8 |
| B200 (Blackwell) | 全部 + FP4 | **FP4/FP8 混合**，FP4 权重 + FP8 KV Cache |

##### 7.2 根据场景选择

```
你的模型能装下吗？
  │
  ├─ 能 → FP16/BF16 推理（最简单，精度最好）
  │
  └─ 不能 → 压缩多少？
             │
             ├─ FP8 就够 → FP8 推理（H100+，最省心）
             │
             ├─ FP8 不够 → INT4 Weight-Only（AWQ/GPTQ）
             │
             └─ INT4 也不够 → FP4 混合精度（B200+）
                                或 考虑模型蒸馏/剪枝
```

##### 7.3 精度验证清单

量化后必须验证精度，不能"压缩了就完事"：

1. **Perplexity 对比**：量化模型 vs 原模型，差距 < 5% 为合格
2. **下游任务评测**：MMLU、HumanEval 等基准，差距 < 2%
3. **长文本测试**：量化对长上下文影响更大，必须单独测
4. **边界 case**：数学推理、代码生成等精度敏感任务
5. **KV Cache 精度**：FP8 KV Cache 在某些模型上可能导致重复生成
