# 写在理解 vLLM 多模态与 LoRA 之前，从"只会读字"到"千面化身"：大模型多模态与适配器的演进简史

> 多模态让模型"长眼睛长耳朵"，LoRA 让模型"一秒换装"——两条看似独立的技术线，在大模型推理引擎中殊途同归。

---

### 引言

想象你经营一家大型咨询公司。最初只有一批"文字分析师"（纯文本 LLM），客户送来的报告他们读得又快又好，但遇到图纸、照片、录音就只能摇头。后来你招了"视觉分析师"和"音频分析师"（多模态），终于能看图听音了——但新问题来了：每个客户行业不同，金融客户要懂合规，医疗客户要懂诊断，法律客户要懂判例。难道每个行业招一批专属分析师？

聪明做法：保留一批"通才分析师"（基座模型），给每人发一套"行业速查手册"（LoRA 适配器）。接待金融客户时翻金融手册，接待医疗客户时翻医疗手册——同一批人，秒切专业。

这就是多模态与适配器在大模型中的角色：**一个让模型感知更丰富，一个让模型专业更精准**。两条技术线各走了五年的演进之路，最终在推理引擎中交汇。本文将按时间线拆解两条线的演进逻辑，为后续深入 vLLM 源码铺好认知地基。

---

### 第一条线：多模态——从"只会读字"到"能看能听"

##### 1.1 起点：文本就是一切（2017-2021）

2017 年 Transformer 诞生后的四年里，大模型的世界只有文字。GPT-1、GPT-2、BERT、GPT-3——它们读文本、写文本、理解文本，对图像和声音一无所知。

这不是疏忽，而是**技术约束**：Transformer 的输入是 token 序列，而图像是像素矩阵、音频是波形序列——它们和文本根本不是一种"语言"。要让模型看图，先得解决一个根本问题：**怎么把图像变成 token？**

```
2017-2021: 纯文本时代

  用户输入: "今天天气怎么样？"
       │
       ▼
  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │ Tokenizer│────→│Transformer│────→│ 文本输出  │
  └──────────┘     └──────────┘     └──────────┘

  能做的: 问答、翻译、摘要、写代码
  做不了的: 看图、听音、看视频
  原因: 图像/音频无法直接 tokenize
```

这就像一家只有文字菜单的餐厅——你能点菜，但看不到配图，不知道菜长什么样。

##### 1.2 第一次突破：视觉编码器 + 桥接层（2021-2023）

2021 年，OpenAI 发布 CLIP（Contrastive Language-Image Pre-training），证明图像和文本可以映射到同一个语义空间。紧接着，Vision Transformer（ViT）证明 Transformer 架构也能处理图像。关键认知转变：

**图像不需要"变成文字"，只需要变成和文字同一维度的向量。**

```
2021: CLIP 的核心洞察

  文本: "一只猫"  ──→ Text Encoder ──→ [0.2, -0.5, 0.8, ...]  ─┐
                                                                    │ 同一空间!
  图像: 🐱的照片  ──→ Image Encoder ──→ [0.3, -0.4, 0.7, ...]  ─┘

  → 图像和文本在同一个向量空间中"距离相近"
  → 模型可以通过向量相似度理解"这张图是猫"
```

2023 年，LLaVA（Large Language-and-Vision Assistant）把 CLIP 的视觉编码器和 LLaMA 的语言模型拼在一起，中间用一个简单的线性投影层（"桥接层"）连接——VLM（视觉语言模型）正式诞生：

```
LLaVA 架构 (2023):

  图像输入                    文本输入
  🐱照片                      "描述这张图片"
     │                           │
     ▼                           ▼
  ┌──────────┐              ┌──────────┐
  │ CLIP ViT │              │Tokenizer │
  │ 视觉编码器│              │ 文本编码器│
  └────┬─────┘              └────┬─────┘
       │                         │
       ▼                         │
  ┌──────────┐                   │
  │ 线性投影  │ ← "桥接层"        │
  │ (MLP)    │   把视觉向量       │
  └────┬─────┘   对齐到语言空间   │
       │                         │
       └──────────┬──────────────┘
                  ▼
          ┌──────────────┐
          │   LLaMA LLM  │  ← 语言模型本体不变
          │   语言模型    │     只是输入多了"视觉 token"
          └──────┬───────┘
                 ▼
          "这是一只橘猫，正趴在窗台上晒太阳"
```

LLaVA 的"桥接层"虽然简单（就是一个 MLP），但它解决了一个根本问题：**语言模型不需要重新设计，只需要在输入端"翻译"其他模态**。这个思路成为后续所有多模态模型的通用范式。

##### 1.3 模态扩展：从看图到听音看视频（2023-2025）

LLaVA 证明了"编码器 + 桥接层 + LLM"的范式可行后，模态扩展水到渠成——每个新模态只需要训练对应的编码器和桥接层：

```
多模态扩展范式:

  图像: CLIP ViT + 视觉投影层 → 视觉 token
  音频: Whisper Encoder + 音频投影层 → 音频 token
  视频: 多帧图像编码 + 时序投影层 → 视频 token

  所有模态的 token 拼接后一起送入 LLM:

  ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐
  │视觉token│  │音频token│  │视频token│  │文本token│
  │ 🐱     │  │ 🎵     │  │ 🎬     │  │ 你好   │
  └───┬────┘  └───┬────┘  └───┬────┘  └───┬────┘
      └───────────┴───────────┴───────────┘
                   │
                   ▼
          ┌──────────────┐
          │   LLM 基座   │  ← 同一个语言模型处理所有模态
          └──────┬───────┘
                 ▼
          "图中是一只猫，背景音乐很舒缓，视频里它在打哈欠"
```

| 模型 | 年份 | 支持模态 | 视觉编码器 | 特色 |
|:---|:---|:---|:---|:---|
| LLaVA | 2023 | 图像 | CLIP ViT | 开创"编码器+桥接+LLM"范式 |
| Qwen-VL | 2023 | 图像 | ViT-bigG | 中文多模态，高分辨率 |
| GPT-4V | 2023 | 图像 | 未公开 | 首个商用多模态大模型 |
| Gemini | 2023 | 图像/音频/视频 | 未公开 | 原生多模态训练 |
| Phi-4-multimodal | 2025 | 图像/音频 | 多编码器 | 小模型多模态 |
| Llama-4 | 2025 | 图像/音频/视频 | 多编码器 | Meta 开源多模态 |

##### 1.4 多模态对推理引擎的挑战

多模态不只是"多加一种输入"，它对推理引擎提出了全新挑战：

| 挑战 | 纯文本时代 | 多模态时代 | 影响 |
|:---|:---|:---|:---|
| **输入预处理** | Tokenize 就行 | 每种模态要独立编码+对齐 | 管线复杂度 ×N |
| **Token 数量** | 几百~几千 | 一张图可能产生 500+ token | 显存和计算量暴增 |
| **序列长度** | 4K~8K 够用 | 图文交织可能 32K+ | 注意力 O(n²) 压力 |
| **模态交织** | 不存在 | 图-文-图-文交替排列 | 批处理难以对齐 |
| **缓存策略** | 文本缓存 | 视觉 token 缓存复用 | KV Cache 管理更复杂 |

> 笔者注：一张 336×336 的图片经 CLIP ViT 编码后产生 576 个视觉 token——相当于一篇短文的长度。多图场景下，仅视觉 token 就可能占掉 KV Cache 的 30-50%。这就是为什么 vLLM 必须从架构层面重新设计多模态管线，而不是"打个补丁"。

### 第二条线：适配器——从"全部重训"到"一秒换装"

##### 2.1 起点：全量微调的黄金时代（2018-2020）

2018 年 BERT 诞生后，"预训练 + 微调"成为 NLP 的标准范式：先在大规模语料上预训练一个通用模型，再在下游任务数据上全量微调。当时模型还小（BERT-base 110M 参数），全量微调不是问题。

```
全量微调 (Full Fine-Tuning):

  预训练模型 (110M 参数)          下游任务数据
  ┌──────────────────┐            ┌──────────┐
  │ ████████████████ │  ─────────→│ 情感分析  │
  │ 全部参数都更新    │            │ 数据集    │
  └──────────────────┘            └──────────┘
         │
         ▼
  ┌──────────────────┐
  │ ████████████████ │  ← 所有 110M 参数都变了
  │ 情感分析专用模型  │     需要存储完整的 110M 参数
  └──────────────────┘

  N 个任务 = N 份完整模型 = N × 110M 参数的存储
```

当模型只有 110M 时，存 10 份也才 1.1GB，完全可接受。但 2020 年 GPT-3 的 175B 参数改变了一切——全量微调 175B 参数需要 1.4TB 显存，存 10 份需要 14TB 存储。**全量微调从"标准做法"变成了"奢侈消费"。**

##### 2.2 第一次突破：Adapter Layers——在模型中插入"小插件"（2019-2020）

2019 年，Houlsby 等人提出 **Adapter Layers**：冻结预训练模型的全部参数，只在 Transformer 层之间插入少量可训练的"适配器模块"。

```
Adapter Layer 思路:

  标准 Transformer 层:
  ┌───────────────────────────────────┐
  │ Self-Attention → FFN → Output    │  全部参数可训练
  └───────────────────────────────────┘

  Adapter 插入后:
  ┌───────────────────────────────────────────────┐
  │ Self-Attention → [Adapter↓] → FFN → [Adapter↓]│
  │                  ↓               ↓            │
  │               ┌─────┐        ┌─────┐         │
  │               │Down │        │Down │         │
  │               │Proj │        │Proj │         │
  │               │(小) │        │(小) │         │
  │               │  ↑  │        │  ↑  │         │
  │               │Up   │        │Up   │         │
  │               │Proj │        │Proj │         │
  │               │(小) │        │(小) │         │
  │               └─────┘        └─────┘         │
  │               冻结部分        冻结部分          │
  └───────────────────────────────────────────────┘

  可训练参数: 仅 Adapter 内部的两层小投影 (通常 < 1% 总参数)
  基座模型: 完全冻结，不更新
```

```
比喻: Adapter 就像给一本百科全书贴便利贴

  百科全书 (基座模型): 175B 参数，冻结不动
  便利贴 (Adapter): 每层贴两张，写上"金融术语补充""医疗知识补充"

  切换任务 = 换一套便利贴，书本身不用重印
  存储: 175B (书) + N × 1.75B (便利贴) << N × 175B (N 本重印的书)
```

Adapter Layers 的参数效率很高（只训练 < 1% 的参数），但有一个致命缺点：**推理时增加额外层，带来延迟**——每个 Adapter 都要过一遍 Down Projection + 非线性激活 + Up Projection，串联在推理路径上。

##### 2.3 第二次突破：LoRA——旁路低秩分解（2021-2022）

2021 年，Edward Hu 等人提出 **LoRA（Low-Rank Adaptation）**，核心思想：与其在模型里插入新层，不如在现有权重旁边加一条"旁路"，用低秩矩阵近似全量微调的权重变化。

```
LoRA 的核心洞察:

  全量微调的权重变化 ΔW 也是低秩的!
  → 可以用两个小矩阵 A × B 近似 ΔW

  原始权重 W (d×d):          LoRA 旁路:
  ┌──────────────┐           ┌──────┐  ┌────────────┐
  │              │           │  A   │  │     B      │
  │   冻结不变   │    +      │(d×r) │×│  (r×d)     │
  │              │           │      │  │            │
  └──────────────┘           └──────┘  └────────────┘
    d × d 参数                  r << d，可训练

  前向传播: y = W·x + (A·B)·x = (W + A·B)·x

  推理时可以合并: W' = W + A·B → 无额外延迟!
```

```
比喻: LoRA 就像给一幅名画加"滤镜"

  名画 (基座权重 W): 175B 参数，冻结不动
  滤镜 (LoRA A×B): 低秩矩阵，只调整画风

  全量微调 = 重画一幅画 (175B 参数全改)
  LoRA = 在原画上叠加一层薄薄的滤镜 (0.1-1% 参数)

  关键优势: 滤镜可以"烘焙"进原画 (W' = W + A·B)
           → 推理时和原画一样快，没有额外延迟!
```

LoRA 的参数量和秩 $r$ 直接相关：

| 秩 $r$ | 可训练参数占比 (7B 模型) | 等效效果 | 典型用途 |
|:---|:---|:---|:---|
| 4 | ~0.02% | 接近全量微调 | 简单风格迁移 |
| 8 | ~0.05% | 接近全量微调 | 通用任务适配 |
| 16 | ~0.1% | 等效全量微调 | 复杂任务适配 |
| 64 | ~0.4% | 可能过拟合 | 高度专业化任务 |

> 笔者注：LoRA 的"低秩"直觉来自一个经验观察——预训练模型在微调时，权重的变化量 ΔW 通常集中在很低的几个维度上。就像一个 1000 维的空间里，微调只在 8 个方向上做了调整——那只需学这 8 个方向，不用学全部 1000 个。

##### 2.4 从 LoRA 到 Multi-LoRA 服务（2023-2025）

LoRA 解决了"怎么高效微调"的问题，但推理服务面临一个新问题：**多个 LoRA 适配器同时在线服务**。

```
Multi-LoRA 服务的挑战:

  传统方案: 每个适配器部署一个独立模型
  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ 基座+金融 │  │ 基座+医疗 │  │ 基座+法律 │
  │ GPU 1    │  │ GPU 2    │  │ GPU 3    │
  └──────────┘  └──────────┘  └──────────┘
  → 3 份基座模型，显存浪费严重

  Multi-LoRA 方案: 共享基座，动态加载适配器
  ┌──────────────────────────────────────┐
  │           共享基座模型 (1份)           │
  │  ┌──────┐ ┌──────┐ ┌──────┐         │
  │  │金融   │ │医疗   │ │法律   │ ← 按需加载│
  │  │LoRA  │ │LoRA  │ │LoRA  │         │
  │  └──────┘ └──────┘ └──────┘         │
  └──────────────────────────────────────┘
  → 1 份基座 + N 个小适配器，显存节省 60%+
```

Multi-LoRA 服务的技术难点在于**批处理**：同一个 batch 里，请求 A 用金融 LoRA，请求 B 用医疗 LoRA——怎么在一个 GPU kernel 里同时计算两套不同的旁路权重？vLLM 的答案是 **Punica 内核**（后文 12 号博客详解）。

##### 2.5 适配器家族全景

LoRA 不是唯一的参数高效微调方法，但它是目前最主流的。一览适配器家族：

| 方法 | 年份 | 核心思路 | 可训练参数 | 推理额外延迟 | 当前使用 |
|:---|:---|:---|:---|:---|:---|
| Full Fine-Tuning | 2018 | 全量更新 | 100% | 无 | 小模型 |
| Adapter Layers | 2019 | 插入新层 | ~1% | 有（串联） | 少 |
| Prefix Tuning | 2021 | 在 KV 前加可训练前缀 | ~0.1% | 有（序列变长） | 少 |
| Prompt Tuning | 2021 | 在输入前加可训练 token | ~0.01% | 有（序列变长） | 少 |
| **LoRA** | 2021 | 旁路低秩分解 | ~0.1% | **无（可合并）** | **主流** |
| QLoRA | 2023 | 4-bit 量化 + LoRA | ~0.1% | 无 | 训练侧 |
| DoRA | 2024 | 分离幅度和方向 | ~0.1% | 无 | 新兴 |

LoRA 能成为主流，关键优势是**推理时零额外延迟**——旁路权重可以合并进基座权重（W' = W + A·B），合并后的计算和原始模型完全一样。其他方法要么增加序列长度（Prefix/Prompt Tuning），要么增加网络层数（Adapter Layers），推理时都有额外开销。

### 两条线的交汇：多模态 + LoRA

##### 3.1 为什么它们注定相遇

多模态和 LoRA 表面看是两个独立问题，但它们在推理引擎中面临同一个核心矛盾：**模型越来越大，但部署预算有限。**

```
矛盾:

  多模态: 模型要处理图像/音频/视频 → 模型更大，显存更紧张
  LoRA:   要同时服务多个专业场景 → 适配器管理复杂，显存更紧张

  交汇点: 多模态模型 + 多个 LoRA 适配器
  → 一个 70B 的 VLM，同时挂载金融/医疗/法律三个 LoRA
  → 推理引擎必须同时处理: 多模态输入管线 + 多 LoRA 动态调度
```

##### 3.2 交汇后的技术挑战

| 挑战 | 纯文本+LoRA | 多模态+LoRA | 难度提升 |
|:---|:---|:---|:---|
| **输入预处理** | Tokenize | 图像编码 + 文本编码 + 模态对齐 | 管线复杂 ×3 |
| **KV Cache 管理** | 文本 token | 文本 + 视觉 token（更长的序列） | 序列长度 ×5 |
| **LoRA 目标层** | QKV/O 投影 | QKV/O + 视觉投影层 | 适配器更大 |
| **批处理** | 同一 LoRA 的请求可以 batch | 不同 LoRA + 不同模态混合 batch | 调度更复杂 |
| **显存管理** | 基座 + N × LoRA | 基座 + 视觉编码器 + N × LoRA | 显存更紧张 |

##### 3.3 vLLM 的交汇方案

vLLM 的设计哲学是**统一抽象**——多模态和 LoRA 不是两个独立模块，而是统一在同一个推理管线中：

```
vLLM 的统一管线:

  请求 A: "描述这张图" + 🐱图片 + 金融 LoRA
  请求 B: "翻译这段话" + 文本 + 医疗 LoRA
  请求 C: "听写这段音频" + 🎵音频 + 法律 LoRA
       │           │           │
       └───────────┴───────────┘
                   │
          ┌────────┴────────┐
          │  多模态预处理管线  │  ← 统一处理图像/音频/文本
          │  (编码+对齐+交织) │
          └────────┬────────┘
                   │
          ┌────────┴────────┐
          │  PagedAttention  │  ← 统一管理 KV Cache
          │  (分页+共享+前缀) │
          └────────┬────────┘
                   │
          ┌────────┴────────┐
          │  Punica 内核     │  ← 统一处理 Multi-LoRA 批量计算
          │  (按请求路由到    │
          │   对应 LoRA 权重) │
          └────────┬────────┘
                   │
                   ▼
          A: "这是一只猫..." (金融视角)
          B: "Translation..." (医疗视角)
          C: "The audio says..." (法律视角)
```

### 四、演进逻辑总结

回顾两条技术线，各自的驱动力和核心突破：

```
多模态演进:

  2021        2023           2024           2025
```

##### 5.2 适配器方法速查

| Prefix Tuning | ~0.1% | 有（序列变长） | 否 | 生成任务 |
